# Домашнее задание к занятию «Базы данных, их типы»

## Задание 1. СУБД

> Кейс 
> Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

> Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?
> 1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.
> 1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?
> 1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.
> 1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?
> 1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.
> 1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?
> 1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.
> 1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?
> 1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

1.1 Для бюджетирования и финансовой аналитики, на мой не искушённый взгляд - лучше всего здесь покажет себя PostgreSQL - надёжно, поддержка сложных запросов и бюджетно. 
Также можно использователь дополнительные расширения, типа OLAP, для прогнозирования рисков.
Если хеширование долгое, можно применить аппаратное ускорение или GPU-хеширование.

1.2 Для лендинга, где требуется скорость, гибкость из-за разнообразии структур данных и частые изменения, лучше подойдут документоориентированные NoSQL , к примеру, MongoDB, которая также подходит для CRM из-за удобства хранения в ней неструктурированных данных о коммуникациях и интеграцией с маркетинговыми инструментами. 

1.3 Для данных задач нужна достаточно простая структура. Это может быть либо иерархическая JSON-база, либо тотже MongoDB, что позволит использовать единую базу данных.

1.4 Для логистики нужна графовая СУБД. Из быстро найденного Neo4j или ArangoDB. И, лучше, не объединять её с отделом закупки, ибо это два разных типа баз данных.
Хотя, в PostgreSQL есть расширения для работы с графами. Но, производительность у них хуже, чем у чистой графовой БД.

1.5 Если смириться с некоторыми недостатками, то можно всё построить на, к примеру, PostgreSQL с расширениями для работы с графами, документами.
Но, если есть возможность, лучше разнести отдельные задачи по разным БД.



> Задание 2. Транзакции
> 2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.
> 2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

1) Начало транзакции.
- система фикрисует время начала транзакции, уникальный ID
- блокирует оба счёта
2) проверка достаточных средств для списания
-  если средств нет, то выдаётся ошибка
3) блокировка средств списанния
4) отправка запроса оператору связа на зачисление
5) подтвержение от оператора
- блокированые средства списываются
- записывается в журнале транзакции сумма и статутс списания
- средства записываются на счёт поступления
6) Запись транзакции
- все изменения становятся постоянные
- генерируется чек

При автоплатеже происходит всё тоже самое, только генерация начала транзакции происходит на основе шаблона.


> Задание 3. SQL vs NoSQL
> 3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.
> 3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

1. Строгая целостность данных (ACID)
SQL гарантирует атомарность, согласованность, изоляцию и долговечность. Это критично для финансовых, банковских и учётных систем, где потеря или дублирование данных недопустимы.

2. Мощный и стандартизированный язык запросов
SQL позволяет выражать сложные аналитические запросы без написания кода на языках программирования.
NoSQL требует императивного кода или ограниченных API.

3. Развитая поддержка связей между сущностями
Реляционные модели оптимальны для данных с жёсткими связями (заказы → товары → клиенты → скидки).
NoSQL документные/ключ-значение плохо справляются с глубокими JOIN-ами без денормализации.

4. Надёжность и предсказуемость в многопользовательской среде
Блокировки на уровне строк, многоверсионность (MVCC) и уровни изоляции (SERIALIZABLE, READ COMMITTED) предотвращают конфликты.
NoSQL часто жертвуют этим ради производительности (Eventual consistency).

5. Зрелая экосистема и инструменты
Средства резервного копирования (PITR), репликации (master-slave, multi-master), мониторинга (pg_stat, profiler), миграций (Liquibase, Flyway), а также интеграция с BI-инструментами (Tableau, Power BI) и ORM (Hibernate, Entity Framework). У NoSQL это часто «самописные костыли».


> Задание 4. Кластеры
> Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.
> На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?

ритерий выбора типа СУБД:
Главный критерий — характер вычислений и модели доступа к данным:

Если преобладают агрегации, аналитика и batch-обработка (например, расчёт выручки по 5 миллиардам транзакций)
→ MPP (Massively Parallel Processing) СУБД.

Если нужны частые точечные обновления с гарантиями ACID (например, счётчики лайков, балансы кошельков)
→ NewSQL (CockroachDB / YugabyteDB) или реплицированные in-memory.

Если данные слабоструктурированы, вычисления итеративные (машинное обучение, графы)
→ NoSQL + Spark или специализированная система (Neo4j для графов, Redis для матричных операций).

Если требуются сложные JOIN и оконные функции на терабайтах
→ Реляционная MPP (Greenplum, ClickHouse для аналитики, Apache Doris).

Лучшая модель распределённых вычислений для 1000 машин:
Модель: Shared-Nothing MPP с колоночным хранением и векторными вычислениями

Почему:

Shared-Nothing — каждый узел имеет свой диск и память. Нет общего хранилища (SAN), которое станет узким местом. При 1000 узлов это единственный способ избежать коллизий.

Горизонтальное партиционирование (sharding / hash + range) — данные распределены по узлам, запросы разрезаются на фрагменты (query fragments), выполняются параллельно на всех узлах, результат собирается (gather merge).

Push-вычислений (push-based execution) — операторы фильтрации, агрегации и проекции «проталкиваются» к данным, а не наоборот. Это минимизирует передачу по сети.

Лучшие кандидаты для такой модели:

Система	Когда выбрать
Apache Spark (SQL + DataFrames)	Если нужны ETL, ML (MLlib), работа с разными форматами (Parquet, ORC, Avro). Победитель по гибкости, но не самая лучшая производительность in-memory для pure SQL.
ClickHouse	Если чисто аналитика (OLAP), данные в основном вставляются, редко обновляются. На 1000 нодах даёт фантастическую скорость агрегаций.
Greenplum (на базе PostgreSQL)	Если нужна сложная аналитика с транзакционной строгостью внутри сегментов и стандартный SQL.
Google BigQuery / Snowflake (cloud-native)	Если не жалко денег и нет желания администрировать 1000 машин. Их движок (например, Dremel/Borca) — эталон MPP.
Trino (formerly Presto)	Если данные лежат в разных источниках (HDFS, S3, MySQL, Kafka) и нужны federated запросы.
Почему не Hadoop MapReduce? — Устарел для интерактивной аналитики. Медленный из-за дисковых швов между map и reduce.

Почему не простой NoSQL (Cassandra, Mongo)? — Не умеют эффективно делать распределённые JOIN и агрегации на 1000 узлов без write MapReduce-кода.

Итоговая рекомендация для 1000 машин:
Для batch-аналитики с сильным SQL-уклоном:
ClickHouse (чистое хранилище) или Apache Spark на HDFS / S3 с Parquet.

Для гибридной нагрузки (OLTP + OLAP) + масштабирование до 1000 узлов:
YugabyteDB (SQL + горизонтальный шардинг) или CockroachDB.

Критерий №1, определяющий выбор:
Частота обновления данных — если редкие вставки → ClickHouse/Greenplum.
Частые обновления и консистентность → NewSQL.
Разнородные вычисления (ML, графы, текст) → Spark.
